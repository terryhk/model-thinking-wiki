# 第11章 设计学：以人为本的问题解决

---

📍 **破除焦虑**：「设计学是设计师的事，和我有什么关系？」→ 设计思维不是「设计好看的东西」，是「站在用户的角度发现问题，创造性地解决问题」——这个能力在任何职业中都有价值。

💡 **一句话心法**：先同理，再定义，再发散，再收敛，再验证——五步循环，以用户反馈为导向。

📊 **学科之眼**：设计思维教我们以人为本、用同理心解决问题。

## 内在联系：态度→方法→工具的三位一体

设计思维虽然在本章独占一个模型，但它绝不是孤立的。它与本书中至少三个核心模型形成了紧密的三角关系：

- **生物学思维（BI-2）提供态度**：接受不确定性，不追求一次做对，而是小步试错、以反馈为导向。没有这种态度，设计思维的迭代就没有止境——你会一直想做到「完美」才肯出手。
- **设计思维（DE-1）提供方法**：五步迭代流程，把「小步试错」变成可操作、可学习、可教授的具体步骤。没有这个方法，生物学思维只是一种哲学——你知道应该试错，但你不知道从哪里开始。
- **MVP（BU-14）提供工具**：最小可行产品，把「快速原型」变成具体的执行标准——什么是「最小」？什么是「可行」？没有这个工具，设计思维的原型步骤没有锚点——你可能会做出太精致的原型（浪费时间），也可能做出太粗糙的原型（无法验证假设）。

而**需求三角（MK-3）**则在这条链的底层提供理论支撑——它告诉你需求由什么构成（缺乏感+目标物+消费能力），设计思维告诉你怎么发现需求的每一个构成要素，MVP告诉你怎么最低成本地验证需求是否真实存在。

这就是为什么设计思维值得独占一章——它是一个枢纽模型，把态度、方法、工具和理论串成了一条完整的链。

---

🔗 **跨学科连结**：设计思维是MVP的哲学基础，也是生物学思维的具体操作方法——三者形成「态度→方法→工具」的三位一体。设计思维的同理心步骤是需求三角中「发现缺乏感」的系统化操作，它的重新定义步骤是博弈论中「换框」策略的产品化版本，它的快速原型理念是工程思维中「先跑通再优化」的设计学表达。

📋 **实战练习**：用设计思维重新定义一个你工作中的问题。不是「如何解决X」，而是「用户真正在烦恼什么」。具体步骤：① 找一个你正在解决的实际问题；② 花30分钟走到用户身边（实地观察或电话访谈），记录3个让你意外的发现；③ 把问题重新写一遍——以「用户如何在___情境下感到___」的格式；④ 发散5个不同方向的解决方案（至少2个是你平时不会想到的）；⑤ 选1个做最粗糙的原型，让1个用户试用并反馈。

📊 **概率工具箱**：设计思维的概率逻辑——每做一次用户测试，你就做了一次贝叶斯更新。测试前，你对用户需求的估计是一个先验分布；每个用户的反馈是一条新数据，你用贝叶斯公式更新你的后验分布。越多测试，你的后验分布越窄、越集中——你对用户需求的概率估计越精确。这解释了为什么设计思维强调「多次迭代」而非「一次做对」：每次迭代的边际信息量递减，但前3-5次测试的边际信息量最大——你从「几乎完全不确定」到「大致确定方向」的跃迁，往往就发生在这几轮里。

💰 **边际成本与机会成本分析**：设计思维的边际成本是「先测试再做」的耐心和时间——一次用户访谈可能需要2小时，一个粗糙原型可能需要半天。但它的边际收益是避免做一个没人要的产品——全面开发一个错误方案的机会成本，可能是数月人力和数百万预算。用MVP测试的失败成本远低于全面上线的失败成本，而设计思维的同理心步骤又进一步降低了MVP失败的概率——因为你从开始就问对了问题。设计思维的投资回报曲线是：前期投入略高（因为要花时间做同理心和原型），但后期回报远超传统做法（因为避开了错误方向的巨大沉没成本）。

```
我用设计思维解决 [我的问题]：

1. 同理心：用户真正在经历什么？
   - 我观察到的3个意外发现：___
   - 用户嘴上说的 vs 实际做的差异：___

2. 重新定义：问题的本质是什么？
   - 原来的问题框定：___
   - 重新框定后：用户在 [情境] 下感到 [感受]，真正需要的是 [需求]

3. 发散：有哪些可能的解决方案？（不评判，先列举）
   - 方案1：___
   - 方案2：___
   - 方案3：___
   - 方案4（最天马行空的）：___
   - 方案5（从别的领域借来的）：___

4. 原型：最小可验证的版本是什么？
   - 我要验证的核心假设：___
   - 最小原型：___
   - 制作时间：___（应 ≤ 1天）

5. 测试：用户的反馈是什么？
   - 用户的实际行为：___
   - 被推翻的假设：___
   - 新发现的洞察：___
   - 下一轮要调整的：___
```
